## Détails techniques

### BUG-063 - Ollama modèle ignoré (chat)
**Problème** : Le BUG-052 (v0.3.6) avait fixé `get_llm_service_for_provider()` (utilisé par le Board), mais le chat utilise `get_llm_service()` → `_default_config()` qui avait un `logger.debug` silencieux. Si la lecture SQLite échouait (conflit avec l'engine async), le modèle retombait sur "mistral-nemo".
**Solution** :
1. `logger.debug` → `logger.warning` pour rendre les échecs visibles
2. Log INFO des préférences lues depuis la DB (provider + model)
3. Log INFO du modèle Ollama effectivement utilisé
4. Timeout SQLite `connect_args={"timeout": 5}` pour éviter les locks avec l'engine async
5. Log WARNING sur le fallback Ollama (quand aucune clé API n'est configurée)

### BUG-064 - Board vierge au premier lancement
**Problème** : `AnimatePresence mode="wait"` (ligne 372 BoardPanel.tsx) jouait l'animation d'entrée (`initial={{ opacity: 0, y: 20 }}`) au premier montage, rendant le contenu invisible. Au close/reopen, l'animation exit/re-enter fonctionnait normalement.
**Solution** : Ajout de `initial={false}` sur l'AnimatePresence enfant pour désactiver l'animation initiale au premier montage.

### BUG-065 - PPTX titre blanc invisible
**Problème** : Le prompt système dans `pptx_generator.py` (ligne 90) disait "Titre de slide : [...] blanc". Le LLM interprétait "blanc" littéralement comme `RGBColor(0xFF, 0xFF, 0xFF)` - invisible sur fond sombre si le background n'est pas correctement appliqué.
**Solution** : Remplacement de "blanc" par "couleur RGBColor(0xE6, 0xED, 0xF7) (text_primary #E6EDF7)" dans le prompt.

### BUG-062b - MCP tool calls timeout
**Problème** : Le fix v0.4.4 (BUG-062) avait augmenté le timeout d'initialisation à 90s, mais `_send_request()` gardait un timeout par défaut de 30s pour TOUS les appels. Les `tools/call` (opérations potentiellement lentes) dépassaient ce timeout.
**Solution** :
1. Timeout par défaut de `_send_request` augmenté de 30s → 60s
2. Timeout spécifique `tools/call` à 120s

### BUG-061b - Email refresh diagnostic
**Problème** : Le fix v0.4.4 (BUG-061) avait ajouté l'enrichissement concurrent, mais si le token Gmail expirait, TOUS les messages enrichis retournaient une erreur, filtrés silencieusement par `.filter(msg => !msg.error)` → liste vide sans explication.
**Solution** :
1. Backend : log WARNING du nombre de messages en erreur vs total après `asyncio.gather`
2. Frontend : `console.warn` des messages en erreur avec le détail de la première erreur
